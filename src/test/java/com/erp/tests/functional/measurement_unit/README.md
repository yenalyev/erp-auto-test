# Functional Tests: Measurement Units (Одиниці виміру)

## Огляд
Цей набір тестів забезпечує повне покриття життєвого циклу та верифікацію цілісності даних для модуля **Measurement Units** у розділі Master Data. Основна мета — гарантувати стабільність словника одиниць виміру, що є фундаментом для ресурсів та складського обліку.

---

## Тестові сценарії

| ID | Назва сценарію | Опис перевірки                                                                                 | Severity |
|:---|:---|:-----------------------------------------------------------------------------------------------|:---|
| **TC-MU-001** | Happy Path: Створення | Успішне створення ADMIN-користувачем, валідація JSON-схеми та перевірка персистентності в БД.  | **Critical** |
| **TC-MU-002** | Валідація даних | Data-driven перевірка обов'язковості полів (Null/Empty) та лімітів довжини (Boundary Testing). | **Normal** |
| **TC-MU-003** | Контроль унікальності | Заборона створення дублікатів (Name/ShortName) та перевірка атомарності операцій.              | **Critical** |

---

## Деталізація функціоналу

### 1. Позитивний сценарій (Happy Path)
* **Contract Validation:** Використання JSON Schema для перевірки відповідності типів даних та обов'язкових атрибутів.
* **Database Sync:** Перевірка того, що об'єкт реально доступний через `GET` запит, а його поля ідентичні даним із `Request`.

### 2. Валідація вхідних даних (Data Quality)
* **Mandatory Fields:** Система має відхиляти запити без `name` або `shortName`.
* **Boundary Testing:** Верифікація обмеження довжини (255 символів). Запобігання помилкам переповнення на рівні бази даних.

### 3. Бізнес-правила та цілісність (Business Rules)
* **Uniqueness:** Перевірка унікальних індексів для полів `name` та `shortName` окремо.
* **Integrity Check:** Унікальна перевірка "незмінності бази". Ми гарантуємо, що при помилці `400` база даних не засмічується "фантомними" записами (кількість об'єктів до та після запиту не змінюється).

## Технічні особливості
* **Паралелізація:** Використання `DataUtils.getUniqueSuffix()` (Timestamp + Thread ID) для повної ізоляції тестів.
* **Звітність:** Allure-кроки розбиті на фази `STEP 1: Arrange`, `STEP 2: Act`, `STEP 3: Assert`.
* **Докази:** До звітів автоматично прикріплюються фактичні JSON-відповіді та очікувані схеми контракту.
